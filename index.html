safari下使用axios觸發timeout

發表於 2018-01-23

最近用express搭mockserver，克服各種網絡環境限制之後，總算一切順利

但是我們的項目基本都是移動端，就要讓手機連上電腦了，這很好辦，同一個局域網就可以了

吃生菜一樣在手機上打開項目網頁，請求數據⋯⋯

Was! 竟然請求不到？？？Aber warum!?

排查

大寫絕望地開始查錯，非常奇怪的是，我看服務器的日誌，接收到參數了，但是偏偏沒有輸出http日誌，看起來response沒有close的樣子

抓包一看，請求確實是成功返回數據了，難道是服務器跨域沒弄好？

試著把跨域設置關掉，噢這下電腦真說你丫沒允許跨域了，那看來跨域設置是成功的

跟跨域沒有半毛錢關係

axios

一臉懵逼地亂點各種請求數據，順手上傳了個圖片，發現竟然傳成功了，電腦也接到了這張照片

因為上傳照片用的是另一套獨立的東西，敢情是axios？好吧那把error alert出來

timeout of 30000ms exceeded

喵喵喵喵喵？？？光陰似箭都不帶一眨眼就30秒過去吧！

關掉

竟然！就順利請求到了！

罪魁禍首竟然是這貨，把所有請求攔截掉了，看了下源碼，依賴 http 和 https 模塊

鑒於這兩坑爹玩意proxy可以各種掛起各種沒響應，我已經沒有什麼是不能接受的了（不管三七二十一黑了再說

坑

  1. safari瀏覽器下，axios + express會因為timeout的設置而攔截了所有請求，ios強制全都使用safari的內核，所以都有這個問題，android是沒問題的。我也不知道為什麼；
  2. 上次也提到過，http和https模塊使用代理的話會導致沒有響應或者 socket hang up，當時試著使用axios，也是同樣的狀況。最後換成unirest就好了；

我的猜測啊，是axios依賴http和https，而unirest依賴的是express，大概是存在什麼兼容問題吧？

safari用axios請求真實環境，無論是代理還是timeout，都沒有出現過問題

最後

用express就搭配unirest吧

萬一像我一樣已經使用axios，那就把timeout先關掉咯

btw抓完包之後不要忘記把手機的代理關掉，有次在家想點外賣，發現app打不開網頁，但手機確實又是可以上網，戀人表示妳手機有毒吧？

後來過了一個月才想起我忘記把代理關掉（。

在項目中期快速使用的mockserver

發表於 2018-01-22

致Root。

畢業之後有人跟我說前後端分離，當時我還體會不到那種姿勢的魅力，覺得不就是兩個人有相對平等的關係，兩個人都有各自的話語權，年輕的我還不知道【分離】兩字的意思，直到有一天出現了一個人（。

【你們應該有各自的床，等啪的時候再上去】

於是我開始買自己的床，但是感覺都不是那麼好睡，要不就是床太小太固定了，適應不了我360種姿勢。所以我決定自己造一張床

上面是不正經的前言，接下來我們正經一點hhhh

就是有人對我說我們要前後端分離，然而根據這個提案實施之後覺得只是前端有了自己的話語權，後端不碰前端代碼，但分離到底分離在哪？沒有本質的感覺。直到有天有個大佬點了我一句：前後端分離應該是有各自的服務器，最後才到環境上一放，微調代碼

這個服務器在前端就是用node.js搭的mockserver，虛擬了後台的接口返回假數據過來，這就避免了我們在開發過程中互相等待的浪費時間的過程，也極大地減少了聯調的功夫，自己與mockserver交互正確的話，在環境上也不會有什麼大的問題

但是找了那麼多mockserver，都覺得不是那麼的好用：

  1. 數據管理的方式不是那麼適應團隊合作；
  2. 擴展性太低，不能適應大部分場景，如：出参入參的加密、跨域、自定義請求頭；
  3. 項目已經做到一半了，轉換成本太高；

既然如此，不如就自己做一個吧

數據管理

在使用mockserver之前，大部分人的假數據都是寫在項目文件中，一方面是冗余不美觀，更過份的是有的項目太早期，代碼根本沒有去註釋混淆，什麼家底都暴露出來給別人看了

建立數據庫不僅對整個團隊有一定的要求，各種安裝培訓都是成本的一部分，每來一個新人還可能要額外教授數據庫使用，還可能因為電腦環境限制不能安裝數據庫

所以何不將假數據保存成一個個單獨的json文件？比起別的方法可能運行效率有所遜色，但適應大部分人的開發習慣，簡單明瞭，最重要的是在團隊中管理就是不用額外安裝什麼

同時以json文件名作為接口名，自動創建接口，省時省力

跨域，自定義請求頭，嗅探

對於沒什麼後台經驗的前端來說，我覺得這是一個世紀大坑，既然已經踩過，那我就先解決好這裡

擴展

想法總是很美好的，然而實際上很多團隊的項目已經執行已久，要加入新工具，面臨的最大問題就是上手轉換成本，比如出参入參有加密，項目中另外寫一個免加密的方法給mockserver，成本不說，風險實在太大了

要如何簡潔便利地對出参入參做擴展，這個問題我認為最好的擴展方法就是不做過多的封裝，畢竟我們是無法兼顧如此之多的項目的各式各樣的複雜需求，所以原汁原味最自由，只要讓大家知道出参和入參在哪，讓他們自己對其進行管理就好，喜歡的話還可以做個分頁

上傳下載

上傳下載功能基本是現在的項目離不開的功能點了，具體沒什麼好說的，就是查文件讀取信息

順帶可以拓展的是，讀取修改json文件，實現PUT和DELETE

轉發

（為了與下面一項區分，這裡叫轉發）

如果要說項目中期引入mockserver最大的阻礙是什麼，那我覺得是原有接口數據

項目做一半了，突然引入mockserver，調了虛擬服務器的接口就調不通真實環境服務器的接口，兩個服務器成了一個對立的關係，有你無我

如果項目的設計是某些必須經過登錄等一系列流程才能進入到某個頁面的single page，那就要命了，把所有接口的出参保存一份下來，聽著我都要報警了

更理想的方法應該是如果在mockserver中找不到這個接口，我們就代理到真實環境中，從真實接口中取數據返回過來

萬事大吉～

代理

僅僅是轉發到真實環境，很多mockserver都考慮到了，但是有的真實項目又有更坑爹的問題：

出於安全保密性等等原因，有的公司的網絡環境做了限制，訪問外網甚至本公司項目的環境都需要通過代理，這個一般是配置在瀏覽器中配置pac文件或代理地址，終端機dns根本就ping不通這個地址，我們需要對轉發再做一次代理

關於這方面，世紀大坑是node.js的 http 和 https 有問題，用agent不是拋出 getaddrinfo ENOTFOUND 就是 socket hang up，更過份的是根本走不進取到response的地方，什麼反應都沒有，想查錯都查不了

如果你自己寫mockserver遇到了這個問題，我同你港，不是你代理地址錯了的問題，改用 unirest 就好

題外話

上面吹完牛逼了，在講代理的時候提到了內外網環境的問題，順帶說一下內網環境怎麼用

因為我們內網環境都是windows，mac我沒試過這樣我就不說了

首先是node安裝：到node.js的官網下一個免安裝的zip文件，拉到內網環境裡，配置下變量。這方面網上資料很多，我不多說

然後是模塊，因為不能npm安裝，只能外網安裝好，然後拖到內網環境，可以直接用。需要注意的是記得打包壓縮，否則會文件名太長

最後

我認為開發項目情景實在不是一個工具能夠涵蓋的，如果發現解決不了自己的問題，那麼不要猶豫，推開自己做吧

就是記得別人踩過的坑不要踩www

原因在於JavaScript的 = 不是【拷貝】，而是【指向】一個內存地址。所以當我直接將 ex = girlFriend 之後，再修改ex，就相當於修改了girlFriend，這就造成了：

  • 換女友的方式是殺了前任；

方法 makeGirlfriend() 的原理是創建一個臨時的對象，然後將需要複製的對象裡的每一個value複製出來，插到臨時對象中，最後返回這個臨時對象——這就是深拷貝。這樣就不再存在指向內存地址相同而導致殺前任的問題。

這個方法的要點是，判斷對象裡的子節點是簡單的數據類型（如：String）還是複雜的（Object）。如果是簡單的數據類型，就直接拷貝；如果是複雜的，就將其另外執行一次深拷貝的方法。

webpack構建多頁面應用的入口管理

發表於 2017-05-11

webpack好用的地方我就不累述了，今天我是想說一下用webpack做多頁面應用，公共文件提取時，入口管理的一點事。

第一次聽這個工具的時候是從當時一個同事那知道的，但他的構建方式就不是那麼自動化，需要手動管理entry和chunk。

如果項目超過幾十個頁面，那將是一個大型車禍現場，而且生成的css文件會變成 common.js.css（

這顯然不是一個好的主意。

目錄結構

首先來看一下目錄結構。

以一個典型的管理平台為例，頁面一般分為三個類型：列表頁、詳情頁、編輯頁，僅僅是這三個類型，如果打包到同一個CommonsChunk裡，你會發現幾乎就沒有什麼公共文件，大費周章最終提取出來的可能就是一些庫啊框架啊之類的基礎文件。

對此我們可以如最開始的代碼所示，將頁面分為多個類型，而這個管理過程，我們不應該通過在開發過程中不斷修改webpack的配置來達成，而應該選擇更便捷的文件夾管理。

始終是一個入口對應一個文件夾，只不過我們在祂們上面再添加一個父級做分類管理，然後通過代碼自動生成entry並根據祂們的歸類放到不同的CommonsChunk做公共文件提取，之後我們只需要堆文件目錄進行管理就可以了。

webpack.config.js

獲取入口和分類

我們需要提取兩樣東西：1. 入口名稱 2. 分類，因此 createEntriesCommons() 接收兩個參數，一個是入口的路徑，一個是類型。

每個入口的名稱作為entries的子對象的key，路徑作為value，以此一一對應起來。

而commons下每個類型為一個子對象，子對象的key是類型名稱，value是一個數組，存放了對應類型的入口。

生成配置

webpackConfig 是webpack的配置，根據自己需要發揮。重要的是：entry 和 plugins 必須先聲明在 webpackConfig 裡。

這樣，webpackConfig的 entry 和 CommonsChunkPlugin() 就生成好了，形式類似本文最開始那個不自動的代碼，唯一的區別在於這裡的 CommonsChunkPlugin() 是通過對象傳參，name不要帶 .js 後綴，生成的css文件就是 common.css 嚕～

最後不要忘記export配置。

可能會遇到的錯

路徑錯誤或者路徑裡找不到文件，導致 entries 或 commons 為空。

最後

大概就是這麼一個管理思路。這裡只做了最簡單的打包提取方案，某些公司對於庫和框架類文件，是合併打包，單獨打包到一起或者使用cdn通過 <script> 來引入，大家看著辦？
